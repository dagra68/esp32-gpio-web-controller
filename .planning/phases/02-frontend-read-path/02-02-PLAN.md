## Context

Plan 02-01 delivered `gpio-ui.js` — the complete frontend. This plan's only code change is one addition to `gpio-controller.yaml`: adding `js_include: gpio-ui.js` and removing `local: true` from the `web_server` block.

**Why remove `local: true`:** ESPHome issue #5704 — `local: true` uses a pre-compiled static index.html that does not inject the `<script src="/0.js">` tag. The result: gpio-ui.js is served at `/0.js` but never loaded by the browser. Removing `local: true` switches to the dynamic index.html that correctly includes the script tag.

**Trade-off of removing `local: true`:** The default ESPHome v3 UI JS will load from `oi.esphome.io` on first visit if not cached. This only affects the ESPHome-provided UI assets — our gpio-ui.js is still embedded in flash and loads from the device. The REST and SSE APIs are C++ server code, unaffected by this change.

**After this plan:** The device at http://10.1.1.162 serves the custom grid. Both the on-device URL and the localhost dev server work. Phase 2 is complete.

## Tasks

### Task 1: Update gpio-controller.yaml — add js_include, remove local: true

The agent makes this change. Read the current `esp32-gpio-controller/gpio-controller.yaml` first, then edit only the `web_server` block.

Current web_server block (from Phase 1):
```yaml
web_server:
  port: 80
  version: 3
  local: true     # Serve JS assets from device flash (offline-capable)
```

Replace with:
```yaml
web_server:
  port: 80
  version: 3
  # local: true removed — incompatible with js_include (ESPHome issue #5704)
  js_include: gpio-ui.js  # embeds gpio-ui.js in flash, served at /0.js
```

All other content in gpio-controller.yaml must remain identical. Do not alter any GPIO entity definitions, api, ota, logger, wifi, or esphome blocks.

Commit: `feat(02-02): embed gpio-ui.js in device flash via js_include`

### Task 2: User reflashes device

This task is user-executed. The agent documents the procedure.

From the `esp32-gpio-controller/` directory (where both `gpio-controller.yaml` and `gpio-ui.js` live):

```
esphome run gpio-controller.yaml
```

This compiles the firmware with gpio-ui.js embedded and flashes over OTA (device must be on WiFi; no USB cable needed after Phase 1's initial USB flash).

Watch the log output for:
- `INFO Successfully compiled` — compile succeeded
- `INFO OTA successful` — flash succeeded
- Device reboots and reconnects to WiFi
- `INFO Connected to ...` line with IP (should still be 10.1.1.162 if DHCP lease held)

If OTA fails (device not reachable): fall back to USB flash as in Phase 1.

### Task 3: Verify on-device serving

User-executed verification. No agent action required.

Open `http://10.1.1.162` in a browser (not localhost — this tests device-served delivery).

Expected:
- Custom pin grid renders with all 11 GPIO cards
- HIGH/LOW badges show current state
- OUTPUT/INPUT labels correct
- No status banner (device serving is same-origin; SSE connects immediately)
- DevTools Network tab: `/0.js` returned with 200, content matches gpio-ui.js

If the default ESPHome v3 UI appears instead (Lit Element component, not our grid):
- Diagnosis: `js_include` is not loading. Verify `local: true` is absent from yaml and reflash.
- Alternative: Open DevTools > Network and check if `/0.js` is requested and returns 200 with our JS content.
